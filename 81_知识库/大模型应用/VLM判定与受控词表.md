# VLM 判定方式 + 受控词表

## VLM 端到端 ≠ 多步 Pipeline

### 反模式：先抽取再匹配

写产品方案时容易把判定逻辑画成：
```
① 语义解析工单/SOP → 提取期望（结构化字段）
② 识别图片对象 → 提取实际（结构化字段）
③ 集合匹配 / 字段比对
④ 输出 是/否
```

读起来逻辑清晰，但**这不是 VLM 的实际工作方式**——VLM 是端到端的多模态模型，一次性接收图 + 文本 + 上下文，直接输出判定，不分阶段执行。

把"步骤"写进方案，会误导：
- 读者以为有 ①②③④ 四个独立服务可调
- 工程实现按 pipeline 拆错
- 「中间结构化字段」实际不存在

### 正确表达：端到端 + prompt 内嵌判定要素

```
输入: [图片 + 工单文本 + SOP 文本 + 标杆样例]
       全部丢给 VLM API

prompt 内嵌「考量要素」:
   • 期望对象（语义层面，不强制结构化）
   • 实际场景（图片层面）
   • 比对规则
   • 输出格式约束

VLM 一次性输出: {label, reason, tag}
```

**判定要素 = prompt 里告诉模型"考量什么"，不是顺序执行的子任务。**

文档表达时把"步骤"改为"判定要素 / 考量项"，去掉序号化动词头。

## 三态判定（不是二态）

四态聚合算法基于**维度级三态**输出：

| 维度判定 | 触发条件 |
|---|---|
| **是** | 全部判定要素一致 且 校准置信度 ≥ 阈值 |
| **否** | 任一要素不一致 |
| **不好判断** | 任一要素证据不足（图糊 / 暗 / 局部 / 无基线 …）|

工单级再按公式聚合：全是 → 通过；任一否 → 不通过；含不好判断 → 待复核。

**三态优势**：让模型「不知道」时有一个**合法出口**，避免硬判。否则模型会在边界 case 上强行猜，准确率反而降。

## 受控词表（Controlled Vocabulary）

模型输出 reason 字段时**不允许自由文本**，必须从预定义词表里挑标签。

### 两套词表 + 命名空间

| 命名空间 | 何时打 | 例子 |
|---|---|---|
| `不通过原因:*` | label=否 | 明显垃圾 / 水渍 / 污渍 / 灰尘 / 杂物 / 手印 / 桶满溢 / 拖痕 / 毛发 / 局部残留 / 其他 |
| `判断难度:*` | label=不好判断 | 照片糊 / 光线暗 / 角度刁钻 / 局部 / 太远 / 遮挡 / 无基线参考 |

### 为什么不自由文本

- 自由文本无法做 bad case 统计 → 没法定位"哪类错误最多"
- 自由文本无法支撑 report 维度分布 → 报表层只能用"不通过率"一个数
- 不同模型 / 不同人标注口径漂移 → 数据可比性差

### 词表的归属

通常挂在**「标杆库 / 判定示范」数据源里**，统一发布、可迭代。

未命中任何标签时用 `其他` + 在 reason 自由文本里写明 — **作为词表扩充信号**（`其他` 占比变高 = 该扩词表了）。

### 实施约束

- 标注与模型输出**只能选取词表中的标签**
- 添加新标签必须先进词表，不能临时自由文本
- 「拿不准」归到最近一类，否则用 `其他`

### 在产品方案里的表达

不要把词表枚举写在方案正文（量大且会变），写成：
> 判不干净 → **否**，按 `不通过原因受控词表` 匹配证据标签

具体词表归 ④ 标杆库 / 判定示范条目维护。
