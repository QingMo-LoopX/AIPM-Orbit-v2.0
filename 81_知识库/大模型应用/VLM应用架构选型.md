# VLM 应用架构选型：调 API vs 自建 RAG

## 核心问题

接入多模态大模型（VLM）完成业务判定时，**调 API + prompt 工程**与**自建 RAG（embedding + 向量库 + 召回）**是两条不同复杂度的路。选错会拖死项目，要做产品方案前先定。

## 选型矩阵

| 维度 | 调 API + prompt 工程 | 自建 RAG |
|---|---|---|
| 工程量 | 轻（封装上下文 + 调一次接口） | 重（embedding 服务 + 向量库 + 召回链路）|
| 调整灵活性 | prompt 可快速改 | 召回 quality 决定判定，调优周期长 |
| 历史样例参照 | 「标杆库」业务字段索引（粗筛）| 向量检索（细到语义相似度）|
| 数据量阈值 | 小到中等 OK | 大数据 + 多样性高才发挥作用 |
| 上线速度 | 快（周级）| 慢（月级）|

## 推荐路径

**第一阶段 — 调 API + 业务数据库存「标杆库」**

标杆库两类条目：
- **视觉基线照**：人工挑选的历史达标范例，按 `空间 × 工单分类` 字段检索；每组 1–3 张
- **判定示范**：复核难例回流的「是 / 否 / 不好判断」典型样例 + 判定说明 + 分类标签，按 `工单分类` 字段检索

调 prompt 时把检索结果塞进上下文。**这一阶段不需要任何向量库**。

**第二阶段（条件触发）— 升级 RAG**

触发条件：
- prompt 调到瓶颈，效果难再提升
- 业务数据量起来了（标杆库 > 数千条）
- 跨项目效果差异大（装修风格 / 区域差异）

升级方式：
- 视觉基线照 → embedding API 编码 → 向量库 + ANN 召回
- 判定示范 → 同上

## 反模式

- **一上来就自建 RAG**：基础设施成本压倒一切，效果反而比 "API + 标杆库" 差
- **完全不准备标杆**：VLM 仅靠 SOP 文字描述判，跨项目装修风格差异大时偏差大
- **把"标杆库"做成 RAG 的简化版**：用 embedding 不是必然要上向量库，简单字段查询就够

## 关键认知

> **向量库是工程基础设施，不是用 VLM 就自动带的能力。**

向量库 vs 数据库 区别：
- 数据库回答"是什么"——精确 / 范围匹配（`WHERE 空间='男卫'`）
- 向量库回答"像什么"——语义最近邻（"和这张图最像的 K 张"）

调 API 模式下两者**配合用**：硬过滤（数据库字段 → 候选缩到同类）+ 向量相似（精筛最像）。**不是替代关系**。
