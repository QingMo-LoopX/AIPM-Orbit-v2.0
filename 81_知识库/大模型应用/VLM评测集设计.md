# VLM 评测集设计

## 核心原则

> **「数据层一起做，任务层独立看」**

- ✅ **数据层一起**：一份评测集、一次标注，同时覆盖所有维度。一个工单一行，行内带多个维度的标注
- ❌ **任务层一起**：把多个维度塞给模型一次性判完、只算一个总分

两层混淆是评测集设计最常见的错。

## 推荐主格式：JSONL

每行一个工单，行内带多维度标注：

```json
{
  "id": "wo_0001",
  "work_order_type": "temporary",
  "scene": "男卫",
  "images": [
    {"role": "before", "path": "..."},
    {"role": "after",  "path": "..."},
    {"role": "report", "path": "..."}
  ],
  "context": {
    "work_order_text": "...",
    "sop_brief": "...",
    "report_text": "..."
  },
  "labels": {
    "object_match":      {"label": "是",    "reason": "...", "tags": []},
    "feedback_complete": {"label": "否",    "reason": "缺事后照片", "tags": ["不通过原因:缺事后"]},
    "clean_pass":        {"label": "不好判断", "reason": "看不全地面", "tags": ["判断难度:局部"]}
  },
  "split": "test"
}
```

## 关键字段填写指南

| 字段 | 由谁产生 | 怎么填 |
|---|---|---|
| `id` | 脚本 | 真实工单号或 `wo_<序号>`，全集不重复 |
| `work_order_type` | 工单数据 | `planned` / `temporary`，决定哪些维度要标 |
| `scene` | 工单 + 归一化 | 用「标准空间形式」（如「男卫」），不是原始「三楼北门男卫」 |
| `images[*].role` | 标注人 | `before` / `after` / `report` 三选一 |
| `images[*].path` | 脚本 | 相对路径 |
| `context.*` | 工单原文 | **照抄不要改写** |
| `labels.<维度>` | 标注人 | 见下方「labels 怎么标」 |
| `split` | 标注人 / 脚本 | `test`（评测）/ `dev`（调 prompt 小样本） |

## labels 三元组规则

每个维度的标注 = `{label, reason, tags}`：

- **label**：三选一 `是 / 否 / 不好判断`。判不了就落「不好判断」，**不要硬判**
- **reason**：一句话自由文本，给人看的，必填
- **tags**：受控词表标签数组（见 [[VLM判定与受控词表]]）

## 标注纪律

1. **维度独立判**，不连坐。一个维度否不影响其他维度
2. **看图与维度对齐**：判 ④ 清洁只看 after；判 ⑤ 报单照只看 report，**不混看**
3. reason 写**维度内**的依据，不要写「这单整体不合格」这种工单级结论
4. 没图可看时给 `不好判断` + `判断难度:*`，不要省略 key 假装满标
5. **不适用的维度整个 key 不写**（如计划工单没 ⑤），不要填空值

## VlmEvalKit 集成模式

```
clean_pass.jsonl（宽表，一工单一行）
                ↓ explode by dimension
                ├→ object_match     → 取 after/before + 套②prompt + 归一化
                ├→ feedback_complete → 取全部反馈图 + 套③prompt + 归一化
                ├→ clean_pass       → 取 after(+before) + 套④prompt + 归一化
                └→ report_photo     → 取 report + 套⑤prompt + 归一化
```

**关键**：源数据是宽表（一工单一行 + 多维度 labels），喂给模型时按维度 explode 成独立评测任务。

VlmEvalKit 那边仍是「一行一题」，每维度导一份 TSV / 喂一批。

**让 VlmEvalKit 当执行器，不要让它定数据结构。** 必要时写 JSONL loader 接进去；纯靠 TSV 会丢失多维度结构。

## 指标分层（不要只看 overall accuracy）

### 维度级（每维度各出一份）
- 否 的召回率
- **误通过率**（人工标否但模型判是） ← **核心红线，每维度都看**
- 不好判断 占比
- 分场景准确率（按 scene）
- bad case 类型分布（按 `不通过原因:*` tag）

### 工单级聚合（评四态逻辑）
- 工单级四态准确率（通过 / 不通过 / 待复核 / 无照片）
- 工单级误通过率（人工判不通过 / 待复核但聚合成通过）
- 维度间错误关联（哪个维度最常拖垮整单结论）

聚合规则沿用产品方案：全是 → 通过；任一否 → 不通过；含不好判断 → 待复核；全是无照片 → 维持。**单独评聚合层**能暴露「单维度都还行、聚合却错」的问题。

## 标签收敛流程

第一阶段词表精简（每命名空间 4-6 个起步），跑 60+ 工单。

等真实 bad case 积累一两百单后：
- 看 `其他` 和 reason 自由文本里反复出现什么
- 把高频的提炼成正式标签 → 进词表 → 之后必须用

这就是**词表收敛**，也是产品方案"持续优化"的工程实现路径。

## 反模式

- ❌ 把 prompt 写死在数据集里 → prompt 调整变成数据集重做
- ❌ 直接把所有信息硬塞进 TSV → 多维度 / 多图 / 元数据没地方放
- ❌ 把维度合并成一道大题让模型一次判完只算总分 → 评不出聚合逻辑
- ❌ 把 VlmEvalKit 当数据源结构标准 → 反过来由工具决定数据结构
