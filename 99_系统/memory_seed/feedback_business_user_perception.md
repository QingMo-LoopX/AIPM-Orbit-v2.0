## 1. 核心原则：权限对业务用户应该是透明的

业务用户（保洁员 / 主管 / 项目经理）打开系统的预期是『顺利干活』，**不会去『查权限』**。
他们应该感知到的是：
- 该看的看到，不该看的不见
- 该干的能干，不该干的连按钮都没有
- 报单 / 转派 → 系统自动路由，业务方不用选『发给谁』
- 项目切换 → 顶部一个入口，能切的项目就在那里

**业务用户应该完全感知不到『权限』这个东西的存在。**
如果他在脑子里想『我有什么权限』——产品设计失败了。

## 2. 方案 / 画板里要区分三层受众

写权限相关方案 / 画板 / PRD 时，措辞要明确**当下在对哪一层受众说话**——不要混着说：

| 层 | 受众 | 关心的 |
|---|---|---|
| **业务用户感知层** | 保洁员 / 主管 / 项目经理 | 我能不能顺利干活；不应该出现『权限』词汇 |
| **运营配置层** | 尹（运营） + 配置人员 | 把谁挂在哪个项目当什么角色、谁是谁的上级 |
| **技术实现层** | 产品 / 技术 / 开发 | 底层数据结构、鉴权逻辑、应用层读写规则 |

## 3. 业务模型在卓越场景里的核心抽象

承载权限事实的结构 = **项目人员页 + 平台级标记**——但这是**给运营 + 产研用的**，不是给业务用户用的。

| 用户类型 | 占比 | 权限事实存在哪 |
|---|---|---|
| 项目级用户（保洁员 / 主管 / 项目经理） | 绝大多数 | 项目配置 → 项目人员页（每行 = 用户 × 项目 × 角色 + 主管） |
| 平台级用户（卓品总部运营） | 少数 | 用户管理 → 账号类型 = 平台级 |

**HR 组织 / 职能岗 / 临时支援 / 区域视图都不存在**——卓越业务里只有上面两类用户。

## 4. 物业业务里组织 = 权限是同一份事实

> 一个用户 + 在哪个项目 + 当什么角色 → 这一句话同时是组织事实，也是权限事实。

『组织树』在物业业务里几乎没有独立价值——降级为 UI 导航层。

## 5. 写作落地

- 痛点 / 现状描述按受众分层写。例如『改组织后上级和数据权限不同步』是运营 / 技术视角；业务用户视角应该是『我调到 B 项目了，但小程序还能看到 A 项目工单』。
- 改造点同样按层写：业务用户感知层 vs 运营配置层 vs 技术实现层 各自的改造目标分开表达。
- 画板设计：可以以三层为纵向骨架，左右仍是现状 / 未来对比；技术实现层的现状不用展开（产研内部已消化）。
