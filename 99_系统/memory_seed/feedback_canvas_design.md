业务画板（`.canvas`）的设计原则：

1. **画板 ≠ 方案文档**
   不要把方案文档的内容（痛点编号、改造点 C1~C11、决策表 D1~D5 等）原样拼成卡片放上画板。方案文档保持 Markdown 形式，画板是另一种表达。

2. **画板要以业务方实际能看到的视图为骨架**
   - 优先用业务方在 PC 端 / 小程序里看到的**菜单 / 模块**作为节点（如组织架构 / 角色权限 / 项目配置 / 用户管理）
   - 不要用产品/技术抽象（如"四层权限模型""归属层""关联层"），业务方对不上
   - 画板上的节点应该让业务方"一眼对到他每天在用的系统"

3. **现状 vs 未来对比布局**
   - 左侧：现状是怎么乱的（用业务实际遇到的问题描述）
   - 中间：箭头/标记"改造方向"
   - 右侧：未来改造后是怎样
   - 在每一侧标出节点之间的关系（耦合 / 授权 / 双入口 等），让业务看见关系链路本身的演变

4. **避免自创业务方没说过的实例**
   不要在画板里编造业务实际不存在的项目示例（如 OPPO）。如要举例，只用会议纪要 / 文档里出现过的真实项目（苏研所 / COE）。

5. **现状描述只讲现象，不展开已被消化掉的设计细节**
   产研内部已经决议要砍掉 / 改造的字段名、技术方案细节，**不要再放在"现状"里**让业务方分心。比如"是否项目组织"字段已决定删除，就只描述"组织架构和项目配置耦合"这个现象，不点字段名。
   对应原则：现状描述应聚焦于**还需要业务方理解 / 推动决策**的事，而不是已经在产研内部消化掉的实现细节。

6. **正式方案/画板里不带"谁谁谁原话"备注**
   `（5.7 李健 / 产品："xxx"）` `（5.7 何 / 技术负责人："xxx"）` `（X 会议原话）` 这种引用考据不要写进正式方案文档或业务画板里——是噪声。
   原话出处留在会议纪要 / memory 里查证就够了；正式产出物里只表达"产品要做什么 / 现状是什么"的结论性陈述。
   `## 二、产研内部共识` 这种章节标题里的"4.23 / 5.7"会议日期、表格里的"出处"列也都去掉，让正文清爽。

7. **不用 § 符号** —— 详见 [写作风格偏好](feedback_writing_style.md)，对话和文档都禁用。

8. **现状属性不要写成"本次改造"动作**
   方案文档里讲"系统现状"时（如"用户层级是平台级 / 项目级两档"），**不要带"删除基线 PRD 的某档"这种语气**——现状就是现状，本次改造也并不是要做这一删除动作。
   只在"本次确实要改"的地方写改造，否则写"系统现状 / 沿用"。

9. **不要把"无问题/已闭环"的事情列为改造点**
   §3.2 核心使用场景表里曾把"角色权限配置 / 角色配置不清晰"当作改造场景——这其实不是配置层的真问题（角色管理菜单本身没问题），如果有问题也是应用层（前端各自硬编码）的事。
   原则：使用场景表 / 痛点表只放**真实存在 + 本次会改的**，不要为了凑齐"3 件事"硬塞。

**Why**：用户两次明确反馈 ——
- 第一次："不是说让你把方案按照画板形式展示，而是内容要更能够被业务理解，拓扑图什么的需要使用更可视的业务视角的画板"
- 第二次："目前的权限关系画板过于简单，不止要说明权限关系，更多是用业务预算说明这几个之间的现状，然后再标注出未来我们的方向"（附 PC 菜单截图作为骨架）

**How to apply**：
- 后续做画板前，先确认骨架（业务能看到的菜单 / 流程 / 操作？）
- 默认采用"现状 vs 未来对比"双栏布局
- 每个节点的描述用"业务方天天看到的字段名"，不用表名 / 字段名 / 抽象层级名
- 不确定骨架时，问用户"画板该用哪个视角作骨架"
