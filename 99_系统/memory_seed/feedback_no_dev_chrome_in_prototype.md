原型给评审 / 业务方看时，要区分两种"看起来像 UI 标签的东西"，处理方式不同：

### 一、UI 内部模式语言 —— 不做成 UI
比如『编辑态』『查看态』『瞬态』『首次复核态』这类**开发/产品内部用来盯状态机的语言**。这些是给自己看的代码层概念，不要做成 UI 徽章或文本透出。模式应当从 UI 可供性本身被读出（表单可编 + 底部『保存』= 在编辑；表单只读 + 底部『编辑』= 在看）。

### 二、版本变更说明徽章 —— 可以保留，但措辞要正确
比如 modal section title 旁的红色徽章、新加筛选项 label 旁的红色小标。这种徽章是给**评审/业务方看『本次原型相比上一版加了什么』**用的，有价值，保留。

但**措辞不能是『新增』『新』这种模糊词**——会被误读成『产品里新功能高亮』（让人以为正式上线后用户也会看到这个红标）。**改成『本次新增』『本期版本新加』之类明确是版本层说明的词**。

**Why:** 2026-06-04 CP07 V4 评审过程中：
- 我先在标题栏挂了『查看态/编辑态』徽章 → 用户反问『不要用编辑态、查看态，是原型需要透出给用户的交互，还是你用来做交底备注的？』
- 我整体审计后建议把 modal section title 旁的『新增』徽章也砍掉 → 用户立刻纠正『A类不砍，新增文案调整为"本次新增"』→ 区分清楚：模式标签不要，版本说明要保留但措辞要准

### How to apply
- 写每一条原型 UI 文案前先分类：
  - 是『当前 UI 处于什么内部模式 / 状态』→ 不写，让 UI 可供性自己表达
  - 是『本次评审 / 本期版本相比上一版加了什么 / 改了什么』→ 写，措辞用『本次新增』『本期改动』
  - 是『业务流程状态』（如待质检/待复核/待反馈/质检完成）→ 写，是用户做事时真的要看的
- 删一个看起来像"交底备注"的东西前先问：它对评审找差异有没有价值？有就留，措辞调清楚；没有就砍
- 切忌一刀切——『所有红色徽章都砍』和『所有红色徽章都留』都是偷懒
