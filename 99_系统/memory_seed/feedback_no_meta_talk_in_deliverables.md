正式交付文档（方案 / PRD / 画板 / 报告 / HTML 可视化版 等）里**不要**出现这类句子：

- **阅读导引**：『读这份方案前』『在开始之前』『首先』
- **元话语**：『先对齐一个认知基线』『下面我们来看』『先做个铺垫』
- **结构自述**：『本方案分 X 部分呈现』『按受众分三层来讲』『接下来分三块讨论』
- **版本变更描述**：『2026-05-13 · 新增 X · 移除 Y · 改成 Z』这类副标题写在 HTML / 文档正文里
- **跨版本对比 / 模型演化说明**：『与 V6 模型的关系：V6 §X 内嵌的『A 模型』属于早期方案，本版升级为『B 模型』。下游字段层与新模型的对齐待人工确认』这类 anno / 注解块也不要写在正文里 —— 上一版长什么样不是这一版读者的事
- **版本链脚注**：『本版 VX.Y · 上一版本：[[xxx-VX.Y-1]]（已归档至 _历史/）』这种脚注也不要写在文档底部 —— 文件命名 + `_历史/` 目录已经表达版本链，正文复述是废话

正确做法：直接以实质内容开头并以实质内容结尾 —— 定义、事实、判断、结论。版本号 + 文件命名 + git/历史目录已经表达"这是第几版""上一版是谁"，正文不要再复述。

**Why:** 正式交付文档的读者是业务方 / 评审，他们要的是事实和判断。导引语 / 元话语只增加阅读成本，不携带信息 —— 这是写作里『元话语 vs 实话』之分，元话语是作者给自己的安全感，实话才是给读者的。2026-05-10 在『组织权限升级产品方案 V1.0.html』背景前提段开头写了"读这份方案前，先对齐一个认知基线："被用户指出是废话。2026-05-13 在『权限关系图-业务视角-V3.3.html』 h1 下面写了"2026-05-13 · 新增『卓越集团』租户层 · 移除『客户实体』抽象 · 用户都挂在租户层"也被指出后续都不需要。2026-05-28 在 XM05『AI质检方案-2026-05-28.html』§3.2 末尾写了"与 V6 模型的关系：V6 §3.1 内嵌的『事前/事后照片对照 + 置信度阈值』属于早期单一维度判定，本版升级为 6 维并行结构化判定"也被用户指出是废话 —— 跨版本对比 / 模型演化说明同样属于元话语，本版方案的读者不需要知道上一版长什么样。

**How to apply:**
- 产出任何 vault 内的方案 / 交付物（HTML / Markdown / Canvas / PPT 文案）后，扫一遍开头和段落首句，凡是阅读导引 / 元话语 / 结构自述都删掉
- 让读者第一行就撞上实质内容
- **对话回复不在此列** —— 对话里适当导引（"先这样，再那样"）可以提高沟通效率，不算违规
