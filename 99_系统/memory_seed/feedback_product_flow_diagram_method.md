凡输出**产品逻辑流程图**，严格按「**节点类型 × 要素槽位**」模型，参考 `00_基线资料/检查单产品方案/drawio.html` 的作图法。

**Why**：2026-06-02 用户在 XM05 AI 质检逻辑流程图迭代中明确这套方法论。V1/V2 drawio 视觉对，但内容堆太满；V3/V4 改成段落式 HTML 文档完全偏离；V5 混合后用户原话『他在每一个环节主要表达的要素是什么』点出来真正的方法论：每类节点装特定要素，规则不堆在节点里、不散落在节点间。检查单方案那张 drawio 是范本。该方法对**所有后续产品逻辑流程图**有效，不限 XM05。

**How to apply**：

## 节点类型 × 要素槽位

- **A 起点 / 动作节点**：谁 + 做什么 + 触发条件
- **B 决策菱形**：判断对象 + 判断条件（**只问不答**，答案由分支线和下游节点承载）
- **C 处理节点**（信息密度最高）：动作 + 核心业务规则 + 边界条件（规则必须写在节点里）
- **D 计算节点**：公式 + 取整方式 + **具体示例**（如 20% × 100 → 起点 1, 21, 41, 61, 81）
- **E 规则判定节点**：触发条件（时间窗 + 数量阈值）+ 命中后状态变换 + 生效范围
- **F 合并 / 集合运算节点**：集合运算公式 + 去重规则 + 命中原因保留方式
- **G 输出节点**：输出物 + 应用端

## 横切要素（独立成块，不入主流程节点）

- **术语定义块**：命名规约 + 关系公式（例：检查覆盖周期 = 检查周期 × 5 次）
- **设计原则块**：为什么这么设计（例：保障覆盖 / 公平轮转 / 避免重叠）
- **异常情况块**：新增 / 删除 / 例外 / 周期重算 — **集中放右上角**

## 取舍原则

1. 主流程节点要短，规则进旁注或处理节点内部
2. 决策菱形只问不答
3. 跨步骤共享规则集中放，不重复
4. 抽象规则必配示例
5. 节点信息密度由类型决定：C/E 最多，B 最少
6. 流程方向自上而下，分支左右展开

## 不要做的

- ❌ 整个流程做成段落式 HTML 长文档（V3/V4 错误）
- ❌ 节点里塞过多无关规则
- ❌ 横切规则散落多个节点重复
- ❌ 抽象规则不给示例
