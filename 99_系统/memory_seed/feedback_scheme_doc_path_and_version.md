方案设计类文档（HTML / Markdown / Word 同样适用）两条铁律：

1. **归属位置 · 铁律**：只要属性是『方案』，**一律**放对应 XM 项目的 `04_方案设计/`。
   - 不分格式：HTML / Markdown / Word / Canvas 都一样
   - 不分阶段：初稿 / 评审版 / 终稿 / 演进版本都一样
   - CP 的 `24_PRD管理/` 只接 **PRD**（产品需求文档），不接『方案』
   - 即使方案最终会回流到 CP 的 1X 架构层（11_架构图 / 13_核心规则），那是『回流』动作，不是把方案放在 CP 的 24_PRD管理 下
   - **不要为方案文档判断『要不要放 CP』** —— 答案恒为否
   - 发现现有方案文档放错位置，先搬迁再改

2. **版本流程**（与 CLAUDE.md『业务文档版本管理』一致，HTML 也算业务文档）：
   - **算结构性修改**（必走版本）：新增章节 / 删除一整个段落或模块 / 改结论 / 加大段内容 / 替换可视化形式（卡片→流程图、表→图）/ 增删一条具体问题项 / 调整核心数据计数（如"3 个问题→2 个"）/ 新增 kpi / 警示块
   - **算小修小补**（可原地改）：错别字 / 链接修复 / frontmatter 补全 / 单字调整 / CSS 颜色微调 / 同义词替换
   - **次小版本 V2.x 适用场景**：在主版本基础上的增量调整，结构变动但未颠覆全局——如 V2.0 加产品流程 → V2.1 把拓扑卡片改成 mermaid 流程图。主结构重写或大段重构起 V3.0
   - **判断口径**：『这次改完，再发给业务方是不是一份新版本？』是 → 走版本流程。任何对图形 / 流程 / 模块结构的替换 = 默认走版本
   - **绝对禁止连续在同一版本上累积多次结构性改动而不归档** —— 哪怕每次改动看起来不大，只要累积到 2 次以上结构变化就必须把版本号往前推

**踩坑案例 2026-05-10**（同一次会话内）：
- 错误模式：V2.0 建好后，连续做了『拓扑卡片→mermaid 流程图』『删一条技术问题』『加一段并集说明』三次结构性修改，全部原地累积在 V2.0 上没建新版本
- 用户两次提醒："为什么又没有新增版本?" / "你改成 V2.1 也行啊"
- 正解：任一改动都应触发版本号推进。即使用户没主动要求，也要主动走版本

(2) 这份文件本就放错目录 —— 在 `CP05_物业系统权限管理2.0_未开始/24_PRD管理/prdword交付版/` 下，本应在 `XM03_权限体系升级_V1_进行中/04_方案设计/` 下。

**How to apply:** 接到对方案文档（任何格式）的修改请求时，先两问：
(a) 路径对不对？放在 XM 的 `04_方案设计/` 还是错放在 CP 的 `24_PRD管理/`？
(b) 是结构性修改还是小修小补？
任一答案需要处理就先做路径搬迁 / 版本归档，再动手改内容。HTML 不是例外。
