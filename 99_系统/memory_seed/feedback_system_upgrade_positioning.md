写**系统级 / 基础设施级**升级方案（权限、组织、鉴权、流程、数据模型等）时，定位上严格区分：

- **方案的目的** = 系统整体能力升级，解决通用问题
- **业务硬节点（如 8/1 苏研所抢单试点）** = 验证场景之一 / 时间窗口，是诸多应用场景的一个，不是方案存在的唯一理由

**修辞上避免**：

- ✗ "XX 改造不是『做更好』，是 X 月 X 日某业务跑起来的前置条件"
- ✗ "X 月 X 日 XX 试点跑不动的直接卡点"
- ✗ 把所有问题都绑到一个具体业务节点上做"放大效应"论证

**修辞上倾向**：

- ✓ "以下问题在多场景下普遍复现，影响所有业务"
- ✓ "X 月 X 日 XX 试点是即将到来的验证窗口之一，方案要解决的是系统整体的 XX 基础设施"
- ✓ 排期可以以业务硬节点为锚（"X 前完成"），但方案目的不绑死在该节点上

**Why:** 把通用问题绑到单个业务节点会带来两个误导：
(1) 让读者以为方案只为这个节点服务，节点过了方案就没必要
(2) 让方案的边界被业务节点的边界限制，错过其他场景的真问题
2026-05-10 用户明确纠正『现存问题』模块：『问题应该是通用问题,并不只针对8.1抢单.8.1抢单只是场景之一,修辞上需要调整,这是针对系统整体的权限升级方案』。

**How to apply:**
- 写问题陈述时：先描述系统通用问题，再列举受影响场景（业务硬节点是场景之一）
- 写动机 / 重要性时：避免『不是 X 而是 Y 的前置条件』这种绑定式修辞
- 写排期 / 里程碑时：业务硬节点作为时间锚可以，但不要在方案目的层捆绑
- 自检口径：『把这个业务节点从方案里拿掉，方案还成立吗？』成立 → 修辞没绑死；不成立 → 措辞绑死了，要改
