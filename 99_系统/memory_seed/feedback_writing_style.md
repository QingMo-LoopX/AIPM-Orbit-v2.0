## 1. 不要出现『§』符号

任何场合下都**禁止**使用 § 符号——包括：
- 方案文档、画板、PRD、纪要等所有正式产出物
- **对话回复里也不能出现**（用户已多次明确要求）

引用章节时，**用『章节名称』包起来**：
- ✅ "见『应用层改造』""本方案『配置层改造 / 数据隔离』"
- ✅ "见第三节""下面那张表"
- ❌ "§六""§3.2""V3 §11"

**Why**：用户多次反馈 §  在视觉上是噪声。第一次提了之后我清掉了文档里的 §，但第二次又在对话里出现 § —— 用户再次明确要求"写在记忆里"。

**How to apply**：写每一条回复 / 每一份文档前先扫一遍是否有 §，发现立即用『…』替换。

## 2. 不带"谁谁谁原话"考据

`（5.7 李健 / 产品："xxx"）` `（X 会议原话）` 这种引用考据不要写进正式文档或画板。
原话出处留在会议纪要 / memory 里查证就够；正式产出物只表达"产品要做什么 / 现状是什么"的结论性陈述。

## 3. 中文标点 / 引号

中文双引号优先用 `『』`（Canvas JSON 里也用）；中文单引号用 `「」`。避免 `""`（除非在 markdown 里需要）。

## 4. 列举要显式标识例子

在文档 / 画板 / 图示里列具体例子时，如果不是穷举，**必须**带 `...` / `等` / `如 ... 等` 标识，避免被误读成『就这些』。

- ✅ `系统员工 / 维修工 / 运营 / ...`
- ✅ `如 系统员工、维修工、运营 等`
- ❌ `系统员工 / 维修工 / 运营`（用 `/` 分隔且无收尾，会被读成穷举）

**Why**：2026-05-13 在权限关系图 V3.3 里写『典型身份: 系统员工 / 维修工 / 运营』，用户指出"还以为就这三个"。哪怕已经用了『典型』前缀，`/` 分隔后没有 `...` 收尾依然会被读成完整清单。

**How to apply**：写任何列举（人群 / 角色 / 场景 / 项目 / 模块 …）时，如果不是『就这些』，结尾必带 `...` 或 `等`；穷举才省略。
