# XM→CP 升级 · 目录与反链

XM 项目升级为 CP 产品时，**CP 不重复建 03_业务建模 / 04_方案设计 目录**。原 XM 的过程产物原地保留，CP 通过反链消费 + 把稳定结论提炼到自己的 1X 架构层。

**Why**：
- vault CLAUDE.md 明确写『XM 项目第一阶段产出（业务建模 / 方案设计）验证完成后，回流到 CP 的 1X 架构层（11_架构图 / 13_核心规则）』
- XM 的 03 / 04 是设计过程档案（多版本 / 讨论稿 / drawio / _历史/），属于历史决策证据，不应该在 CP 再复制一份——两份不同步反而带来维护负担
- CP 应该承接的是『稳定提炼』（架构图 / 核心规则），不是『过程档案』

**How to apply**：
- 给 CP 建骨架时**不要**创建 03_业务建模/ 04_方案设计/ 目录，CP 骨架是 `00_会议纪要 / 11_架构图 / 12_路线图 / 13_核心规则 / 21~25`
- XM 改名 `XMxx_xxx_已转化/`，folder note 加 `状态: 已转化` + `转化为: "[[CPxx_xxx]]"`，主体内容**原地保留**
- CP folder note 写一句『方案设计源头：[[XMxx_xxx_已转化]] 的 04_方案设计/』，让读者能跳回去
- CP 的 11_架构图 / 13_核心规则 里**只放稳定提炼**（如状态机定稿 / 核心判定规则），内容里反链回 XM 对应方案版本
- 后续若 CP 进入新一轮升级（如 V2），起一个新 XM，又走一轮『业务建模 + 方案设计』，验证完成后再沉淀回 CP 的 1X

**反例（不要做）**：
- ❌ CP 下复制 XM 的 04_方案设计/ 全部文件
- ❌ CP 下另起一个『方案设计/』目录（命名不一样但语义重复）
- ❌ 把 XM 的方案 HTML 物理搬到 CP/11_架构图 下当架构图（HTML 是设计稿，不是稳定结论）
