# 产品与项目结构总纲

vault 内 CP（产品）/ XM（项目）的统一结构规范。核心原则：**一套骨架通用，类型只影响 CP 的工件槽位，XM 永远通用。**

## 〇、统一模型

```
骨架（所有 CP / XM 通用）：00 流式 / 1X 沉淀 / 2X 流水线 / 3X 运营
        │
唯一变化点：CP 的「类型(软件 / 服务)」决定 1X + 2X 的工件槽位
        │
XM：永远通用，类型靠 frontmatter `产品: [[CPxx]]` 继承
```

→ 全系统只有 **软件 CP / 服务 CP / 通用 XM** 三种形态，不存在「软件 XM、服务 XM」。

判断要不要为某类型新增结构，只看一条标尺：**两个类型需要不同的槽位 → 拆；只是同槽位内容不同 → 通用。** CP 装资产（软件出架构/原型/PRD/测试；服务出矩阵/方法论库/获客/交付），槽位本身不同 → 拆。XM 装「想清楚」（建模 + 方案），槽位一样只是内容不同 → 通用。

## 一、CP 按类型分叉

软件型产品 = 交付物是软件 / 系统，长期迭代维护；服务型产品 = 交付物是人工服务（诊断 / 咨询 / 陪跑 / 工作坊 / 训练营类），按交付批次推进。

| 层 | 软件 CP | 服务 CP | 这层回答什么 |
|---|---|---|---|
| 流式 | 00_会议纪要 | 00_客户与会议 | 高频入口：访谈 / 销售 / 交付纪要 |
| 1X 沉淀 | 11_架构图 | **11_服务产品矩阵** | 卖哪几款 / 各多少钱 / 买了拿啥（SKU 定义） |
| | 12_路线图 | 12_路线图 | 首发方向 / 演进 / 扩张 |
| | 13_核心规则 | 13_核心规则 | 定价 / 对外口径 / 不可承诺 |
| | — | **14_方法论库** | 可复用 IP（护城河） |
| 2X 流水线 | 21_原始需求 | **21_对标与立项** | 为什么做（pipeline 起点） |
| | 22_需求卡片 | **22_获客内容** | 怎么来流量 |
| | 23_原型管理 | **23_销售物料** | 怎么成交 |
| | 24_PRD管理 | **24_交付SOP** | 怎么不靠创始人交付 |
| | 25_测试用例 | **25_交付物模板** | 客户最终拿到什么 |
| 运营 | （40_线上问题） | **31_交付实例** | 每个真实客户一份 → 复盘回流 14 |

**服务 CP 关键差异点：**
- 软件 CP 的护城河在 `11_架构图`；服务 CP 的护城河在 **`14_方法论库`**——「卖产品」而非「卖时间」的核心资产，单列。
- 软件 2X 是「需求 → 代码」；服务 2X 是**客户旅程「获客 → 成交 → 交付」**，其中 `24_交付SOP` 对应目标「别人也能开、不靠创始人出场」。
- `31_交付实例` 的复盘**回流 `14_方法论库`**——每交付一次 IP 厚一层，这是服务产品复利的地方。

### 服务 CP 目录结构

```
20_产品/CPxx_xxx_{未开始|运营中|已下线}/
├── CPxx_xxx.md              # folder note（类型: 服务）
├── 00_客户与会议/           # 访谈 / 销售会 / 交付会 纪要
├── 11_服务产品矩阵/         # 卖哪几款 / 各多少钱 / 买了拿啥（引流款·利润款·承接款）
├── 12_路线图/               # 首发方向、版本演进、扩张计划
├── 13_核心规则/             # 定价规则、对外脱敏口径、不可承诺事项
├── 14_方法论库/             # 诊断框架 / 模型 / 罗盘（长期沉淀的 IP）
├── 21_对标与立项/           # 商业模式诊断、对标、市场判断
├── 22_获客内容/             # 引流文、获客文、产品页、SEO
├── 23_销售物料/             # 报价单、方案书、服务说明、FAQ、异议处理
├── 24_交付SOP/             # 交付流程、checklist、排期 / 分工模板
├── 25_交付物模板/           # 诊断报告 / 路线图 / 交付清单 模板
└── 31_交付实例/             # 每个真实客户一份（脱敏）+ 复盘 → 回流 14
```

> 只建有内容的目录，空槽位有内容再长（不过度）。软件 CP 结构见下方「软件 CP 目录结构」或 README。

## 二、XM 通用结构

XM 是上游「想清楚」层，类型无关，所有项目同一套：

```
10_项目/XMxx_xxx_{进行中|已完成}/
├── XMxx_xxx.md             # folder note（frontmatter 必填 产品: "[[CPxx]]"，独立项目写 产品: 无）
├── 00_项目会议/
├── 01_原始需求/
├── 02_需求卡片/
├── 03_业务建模/
└── 04_方案设计/
```

- **类型继承**：挂在软件 CP 下 = 软件项目，挂在服务 CP 下 = 服务项目，不需要给 XM 加类型字段。
- **XM 的「终」= 业务建模 + 方案设计完成**，不在 XM 内出 PRD / 原型 / 测试。这些回流到所属 CP 的对应槽位（软件 → 21~25；服务 → 22~25），第一阶段成果回流 CP 的 1X 沉淀层。

03 / 04 唯一的类型感知（写什么的内容差异，不是结构差异）：

| 阶段 | 软件项目 | 服务项目 |
|---|---|---|
| 03_业务建模 | 数据模型 / 流程 / 状态机 | 客户旅程 / 交付流程 / 角色分工 |
| 04_方案设计 | 技术方案 / 架构选型 | 服务设计 / SOP 草案 / 定价模型 |

## 三、`11_服务产品矩阵` 模板

服务产品矩阵 = 这个产品对外卖的几款 SKU 清单：卖给谁 / 多少钱 / 买了拿啥。一变（砍款 / 加款 / 改价），下游 `22 获客 / 23 销售 / 25 交付物 / 14 方法论库` 全跟着变，所以归在沉淀层 1X。

典型三层组合：

| 款式 | 典型形态 | 客户拿到什么 | 作用 |
|---|---|---|---|
| 引流款 | 固定价诊断 / 认知课 | 诊断报告 + 路线图 | 拉新、建立信任 |
| 利润款 | 定制交付 / 季度陪跑 | 定制交付物 | 赚钱主力 |
| 承接款 | 方法论库会员 / 订阅 | 持续可用的方法论 | 留住客户、复购 |
