## 项目内部结构

`10_项目/XMxx_xxx/` 下：
- folder note：`XMxx_xxx.md`
- `00_项目会议/` 高频入口
- pipeline：`01_原始需求/ → 02_需求卡片/ → 03_业务建模/ → 04_方案设计/`
- frontmatter 必填 `产品: "[[CPxx_xxx]]"` 声明所属产品（独立项目可写 `产品: 无`）

> **XM 项目的"终"= 业务建模 + 方案设计完成**，不在 XM 内出 PRD / 原型 / 测试用例。
> 这些 2X 工件（原始需求 / 需求卡片 / 原型 / PRD / 测试）统一沉淀到所属产品 [[CPxx_xxx]] 的 `21~25` 工件 pipeline，由产品形态承接长期演进。
> XM 项目第一阶段产出（业务建模 / 方案设计）验证完成后，回流到 CP 的 1X 架构层（11_架构图 / 13_核心规则）。

## 默认工作流

1. **分流原始素材**——根据归属判断目的地：
   - 历史 / 现行 baseline → `00_基线资料/CPxx/...`
   - 新提出的想法 / 诉求 → `00_想法&诉求池/`
   - 明确归属某产品 → `20_产品/CPxx/21_原始需求/`
   - 明确归属某项目 → `10_项目/XMxx/01_原始需求/`
   - 通用方法论 → `81_知识库/`
   - 研究类 → `80_研究沉淀/`

2. **想法演化**——从 `00_想法&诉求池/` 演化为：
   - 项目（有始有终）→ `10_项目/XMxx_xxx/`
   - 产品（长期运营）→ `20_产品/CPxx_xxx/`
   - 消亡 → `00_想法&诉求池/99_归档/已消亡/`
   - 转化时主体内容移到目标位置，原想法留指针标 `状态: 已转化`

3. **工件流水线**：
   - **产品级（CP）**：原始需求 → 需求卡片 → 原型管理 → PRD管理 → 测试用例（21 → 25）
   - **项目级（XM）**：原始需求 → 需求卡片 → 业务建模 → 方案设计（01 → 04）
   - XM 项目方案验证完成后，业务建模 / 方案设计成果回流到所属 CP 的 1X 架构层
   - 每份工件 frontmatter 必填 `产品`，用 `[[wiki-link]]`
   - 下游工件 wiki-link 回链上游

## 命名规范

- 产品编号：`CP{NN}`（如 `CP01`, `CP02` ...）
- 项目编号：`XM{NN}`（如 `XM01`, `XM02` ...）
- **编号永不复用**——即使消亡也保留编号，避免引用断裂

## Folder Note 模式

文件夹与其概览 .md **同名**：
- `CP01_xxx/CP01_xxx.md`
- `XM01_xxx/XM01_xxx.md`

装 [Folder Notes 插件](https://github.com/LostPaul/obsidian-folder-notes) 后，点击文件夹直接打开概览。

## Frontmatter 约定

```yaml
---
产品: "[[XX]]"             # 关联类字段必须用 [[wiki-link]]，不用纯字符串
类型: 项目 / 产品 / 想法 / PRD / 业务需求 / ...
状态: 进行中 / 已完成 / 已修复 / 评估中 / 已转化 / 已消亡 / ...
更新日期: YYYY-MM-DD
---
```

frontmatter 后**不空行**。

## Rules

- **中文输出**（除非用户明确指定外语）
- **先结构化，再生成正文**
- 所有 PRD / 原型 / 测试 frontmatter 必填 `产品`，并通过 `[[wiki-link]]` 回链上游
- 高风险结论显式标记 `**待人工确认**`
- frontmatter 关联类字段（产品 / 上级项目 / 关联PRD / 涉及迭代等）一律使用 `[[wiki-link]]`
- AI 起草、补全、提示风险；人决定范围、优先级、最终定稿

## Slash Workflows

- `/收集原始需求`：把外部输入分流到对应位置（基线 / 想法池 / 产品 / 项目）
- `/生成需求卡片`：把原始需求提炼为结构化卡片（问题/目标/范围/风险/依赖/待确认项）
- `/补原型`：从需求卡片生成原型管理
- `/写PRD`：从需求卡片和原型生成 PRD（含数据模型、页面流程、上线计划、验收标准）
- `/生成测试用例`：从 PRD 生成测试点 / 测试用例 / 验收标准
